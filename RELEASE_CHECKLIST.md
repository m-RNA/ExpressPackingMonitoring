# 发布前核心业务与恢复检查

每次正式发布、生成完整包或增量包前，建议完成本清单。自动化检查通过不代表实机恢复场景已经通过；尚未完成的实机场景应在发布结果中明确说明，但不阻止打包。

## 自动化门禁

- 运行 `pwsh -NoProfile -File Tools/Test-Release-Automated.ps1`
- 确认隔离 WPF 启停、userscript 并发/延时/多监控端和 Web 播放/剪辑界面自动验收全部通过
- 确认必需的核心测试均存在，没有被删除或改名绕过
- 确认 Release 全量测试全部通过
- 确认 Web JavaScript 语法检查通过
- 确认完整解决方案 Release 构建为 0 警告、0 错误

## 扫码、录像与摄像头恢复

- 发货模式扫码后立即开始录像，再扫同码可以正常停止并保存
- 退货模式扫码后立即开始录像，录像、订单模式和扫码动态均正确
- 连续扫描不同单号时，前一录像正常收尾，后一录像正常开始且单号不串
- 摄像头稳定识别面单后自动开始录像，录制中识别新面单能够切换录像，同时扫码枪仍能正常开始、切换和停止
- 开启同码停录后，摄像头必须先等待原单号消失（默认 3 秒），重新出现后在确认窗口内（默认 1.5 秒）累计识别 3 次才停录；持续留在画面内、只识别 1 至 2 次或超过窗口不得误停
- 等待摄像头自动休眠，直接扫码唤醒；预览必须恢复，同一次扫码必须真正开始录像并生成有效文件
- 摄像头休眠后通过鼠标或键盘唤醒；预览必须恢复，随后扫码必须正常录像
- 两个高峰不休眠时段默认均为空；配置时段后，摄像头达到原空闲超时时在时段内保持运行，离开时段后能够按原超时规则休眠
- 空闲时拔出并重新插入摄像头；预览恢复后扫码能够录像
- 录像中断开摄像头；当前录像按规则保存或明确失败，重连后下一单可以正常录像
- 模拟预览冻结或相机无帧；软件必须重连或明确报警，不允许按钮显示录制但没有实际文件
- 关闭程序时有正在录制的订单；退出保存完成，重启后录像和扫码动态可恢复

## 文件、数据库与异常恢复

- 强制结束一次录制进程，重启后残留 MKV 能按规则恢复或保留，并且不会丢失数据库记录
- 制造过短、过小录像，确认文件和数据库删除状态一致，扫码动态不会保留无效记录
- 模拟主编码器失败，确认 CPU 回退成功或明确报警，不能产生假成功录像
- 启用音频时模拟麦克风失败，确认保留策略、日志和提示正确，不生成伪装为正常的无声成品
- 模拟存储目录不可写和空间不足，确认拒绝开录或按预留空间策略清理，不损坏其他目录文件
- 重启软件后只恢复当天最近 20 条有效扫码动态，不恢复已删除、未结束或前一天记录

## 退款报警与打印端降级

- 发货、退货模式扫描退款订单均报警，关闭报警开关后均不报警
- 退款状态文案、工业警报和一次语音播报正确，普通提示不能覆盖退款警告
- “播报音量最大”默认开启；播报前能够取消静音并调至最大，关闭后不得修改系统音量
- 普通打印页面打开或订单变化时可以自动推送订单信息，不依赖退款核验工作页，不需要点击打印
- 退款核验工作页在后台自动打开且不抢焦点，标题和半透明遮罩清晰可辨，只有该页面会切换“打印后退款”筛选
- 关闭退款核验工作页后能够按冷却规则自动重建；用户正在操作的打印页面不切筛选、不刷新、不被遮罩
- 打印端在线时先返回当前退款列表；扫描不在列表中的历史退款单号时，能够按快递单号精确查询并返回正确状态
- 5 秒内快速扫描多个订单只发起一次请求，但所有扫描单号都完成缓存检查和新结果复查
- 打印端离线或超时时使用 SQLite 中最近 90 天的订单数据检查，录像仍立即开始
- 打印端重新上线后能够恢复查询，延迟结果不会改变后来订单的录像状态
- 新快照的时间早于数据库记录时，不得用旧快照覆盖较新的退款状态
- 升级旧版本后 `orderinfo_cache.json` 能迁移到 SQLite 并删除；过期 90 天或超过 50000 条的订单缓存按规则清理
- 重复单号检查覆盖最近 30 天的未删除录像记录，跨重启仍然有效
- Web 访问密钥开启和关闭时，订单推送及退款核验接口均能正常工作
- 从安装向导安装的脚本包含当前监控地址的精确 `@connect` 权限；浏览器不应反复询问同一工位的跨源访问权限

## Web、升级与发布产物

- Web 播放、下载、单界面剪辑、选区播放、生成并下载均正常
- 局域网服务关闭再开启、端口重启后可以重新访问
- 完整包不包含配置、数据库、日志、录像和其他本机运行状态
- 完整包包含预生成的默认 Edge TTS 语音缓存，首次使用固定文案不需要现场生成
- 增量包不包含 TTS 缓存，并验证补丁清单、`update_vX.Y.Z.json`、启动器清单、标签和程序版本号一致
- 增量包根目录包含 `双击安装增量更新.cmd`、`apply_app_patch.ps1` 和 `增量更新说明.txt`，完整解压后可根据 `config.json` 中的 `AppRootDirectory` 定位原 `app` 目录
- 使用双击安装脚本从旧版本升级一次，确认运行中的程序能够正常退出、文件校验成功、失败路径能够回滚，配置、数据库和录像均保留
- `update_vX.Y.Z.json` 的更新内容与最终发布说明一致，合并发布时包含尚未正式发布版本的有效改动
- GitHub 上传完整 zip、更新 JSON 和可用的 AppPatch，默认不上传启动器清单和发布信息文件
- Gitee 上传更新 JSON 和可用的 AppPatch，不上传完整 zip、启动器清单和发布信息文件；仅打开新建 Release 页面，由用户手动填写并上传
- 使用发布包执行一次启动器自动升级，确认配置、数据库和录像保留，启动器可以正常进入应用

完成以上实机检查后，打包时可传入 `-ConfirmManualCoreChecks` 记录确认状态。未传入时打包脚本会给出警告并继续，发布结果需说明尚未验证的实机场景。
